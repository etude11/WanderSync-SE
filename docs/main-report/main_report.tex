\section{Project Overview}

WanderSync is a web-based travel management platform that brings together three things most
travel apps handle separately: booking aggregation, real-time disruption detection, and social
trip coordination. The core idea is simple. When you are travelling, your flight details are in
one app, your hotel is in another, and your ground transport is in a third. When something goes
wrong --- a delay, a cancellation, a storm --- you have to manually check each of them and figure
out what to do. WanderSync replaces that with one timeline that updates automatically and tells
you what changed and what to do about it.

\subsection{What the Prototype Demonstrates}

The prototype implements the full disruption pipeline end-to-end: a user registers, creates an
itinerary, adds bookings, and then receives a real-time disruption alert pushed directly to their
browser the moment the system detects a problem. The alert includes suggested actions generated
by a rule-based engine backed by the Gemini LLM. The itinerary page and disruption page are both
fully functional; the social feature is scaffolded at the data model level.

\subsection{Architecture Decision Records}
% ----------------------------------------------------------------

All four ADRs follow the Nygard template: Status, Context, Decision, Consequences.

\subsubsection{ADR-001 --- Event-Driven Architecture for Disruption Detection}

\begin{mdframed}[backgroundcolor=adrbox, linecolor=accent, linewidth=0.8pt]
\adrfield{Status}{Accepted}
\end{mdframed}

\textbf{Context.} Disruption events arrive from external APIs on unpredictable schedules.
Connecting the Disruption module directly to the Itinerary module with a synchronous call would
couple them tightly --- a slow API response would block itinerary updates, and adding a new
consumer (like email notifications) would require changing the Disruption module.

\textbf{Decision.} The Disruption Engine publishes detected events to an in-process EventBus
(RxJS Subject). The Itinerary and Notification modules subscribe independently. No direct
call exists between them.

\textbf{Consequences.}
\begin{itemize}[leftmargin=1.5em, itemsep=1pt]
  \item \textbf{+} Adding a new consumer requires zero changes to the Disruption module.
  \item \textbf{+} Delivery latency from detection to client browser is $\sim$71 ms (in-process
    dispatch + SSE frame).
  \item \textbf{--} The in-process EventBus is invisible across server instances; sticky sessions
    or Redis Pub/Sub are needed when scaling beyond one process.
  \item \textbf{--} Async flows are harder to trace without structured logging.
\end{itemize}

\subsubsection{ADR-002 --- API Gateway as Single Entry Point}

\begin{mdframed}[backgroundcolor=adrbox, linecolor=accent, linewidth=0.8pt]
\adrfield{Status}{Accepted}
\end{mdframed}

\textbf{Context.} Without a central entry point, every module would need its own JWT validation,
CORS headers, and rate limiting. That is a lot of duplicated security logic, and any module that
misses it becomes a gap in the security perimeter.

\textbf{Decision.} All external traffic enters through a single NestJS global prefix
(\texttt{/api/v1}) with a global \texttt{ValidationPipe} and JWT guard. Backend modules are not
directly reachable by clients.

\textbf{Consequences.}
\begin{itemize}[leftmargin=1.5em, itemsep=1pt]
  \item \textbf{+} Security policy is enforced in one place; impossible to add a route that
    bypasses it accidentally.
  \item \textbf{+} One base URL for all clients; adding a new module does not change any
    client-side URL configuration.
  \item \textbf{--} The gateway is a single point of failure for all user traffic.
  \item \textbf{--} Fine-grained resource ownership checks (``does this user own this
    itinerary?'') must stay in each domain module --- the gateway cannot do this.
\end{itemize}

\subsubsection{ADR-003 --- Adapter Pattern for Booking Providers}

\begin{mdframed}[backgroundcolor=adrbox, linecolor=accent, linewidth=0.8pt]
\adrfield{Status}{Accepted}
\end{mdframed}

\textbf{Context.} AviationStack, hotel APIs, and transport providers each return data in a
completely different schema. If parsing logic for each provider were scattered through the
Booking module, adding a new provider would be a surgery project through multiple files.

\textbf{Decision.} Each provider gets one Adapter class that implements a shared
\texttt{BookingStrategy} interface. The Booking module only ever calls the interface; it has no
idea what the underlying provider returns.

\textbf{Consequences.}
\begin{itemize}[leftmargin=1.5em, itemsep=1pt]
  \item \textbf{+} Adding a new provider means writing one new Adapter class and registering it
    --- no changes to any existing code.
  \item \textbf{+} Each adapter can be unit-tested in isolation with mocked HTTP responses.
  \item \textbf{--} For a single provider, the abstraction is overkill. The pattern pays off at
    provider two and beyond.
\end{itemize}

\subsubsection{ADR-004 --- Stateless JWT Authentication}

\begin{mdframed}[backgroundcolor=adrbox, linecolor=accent, linewidth=0.8pt]
\adrfield{Status}{Accepted}
\end{mdframed}

\textbf{Context.} Server-side sessions require shared session storage (a database or Redis) that
every module must be able to read. For a system that may eventually run as multiple processes,
that creates a scaling dependency. Stateless JWT tokens carry identity in the token itself, so
any server instance can validate them without a shared store.

\textbf{Decision.} Issue signed JWTs on login (24-hour expiry, HS256). The JWT guard validates
the signature and expiry on every request using the shared secret. Passwords are hashed with
bcrypt (salt rounds = 10).

\textbf{Consequences.}
\begin{itemize}[leftmargin=1.5em, itemsep=1pt]
  \item \textbf{+} Any server instance can validate a token without a round-trip to a session
    store --- scales horizontally with no changes.
  \item \textbf{+} Passwords are never stored in plain text; the hash is one-way.
  \item \textbf{--} Tokens cannot be invalidated before expiry (no server-side revocation list).
    A stolen token is valid for up to 24 hours.
  \item \textbf{--} No refresh token implemented in the prototype; users must re-login after 24
    hours.
\end{itemize}

\section{Task 3: Architectural Tactics and Patterns}

% ----------------------------------------------------------------
\subsection{Architectural Tactics}
% ----------------------------------------------------------------

Tactics are specific design decisions that make a quality attribute easier to achieve.
The five below map directly to the NFRs from Task 1.

\begin{longtable}{@{}L{3cm}L{2.5cm}L{7cm}@{}}
\toprule
\textbf{Tactic} & \textbf{NFR Addressed} & \textbf{How it works in WanderSync} \\
\midrule
\endhead

\textbf{Caching} &
  NFR1 -- Performance &
  The Itinerary module stores assembled timeline objects in Redis, keyed by itinerary ID.
  Requests hit the cache first; the database is only queried on a cache miss or after a
  booking mutation invalidates the entry. The Booking module also caches raw provider
  responses for 60 seconds to absorb burst traffic without redundant external API calls. \\[0.5em]

\textbf{Retry with Circuit Breaker} &
  NFR1, NFR2 &
  Every external API call (AviationStack, OWM) retries up to two times with exponential
  backoff (500\,ms, 1\,s). After three consecutive failures in a 30-second window, the circuit
  opens and the system falls back to cached data or marks the provider as temporarily
  unavailable with a user-visible status indicator --- no crash, no unhandled exception. \\[0.5em]

\textbf{Authenticate at the Gateway} &
  NFR3 -- Security &
  A NestJS global JWT guard intercepts every request before it reaches any module. TLS is
  terminated at the gateway. Backend modules receive only internally forwarded requests with a
  decoded identity header (\texttt{X-User-Id}, \texttt{X-User-Role}). Fine-grained resource
  ownership checks remain in the domain modules because the gateway cannot know whether a user
  owns a specific itinerary. \\[0.5em]

\textbf{Event Queue with Persistence} &
  NFR2 -- Reliability &
  The Disruption Engine publishes events to Redis Streams with consumer group semantics. If the
  Notification consumer is briefly offline, it replays missed events from the stream on restart.
  The persisted log is also the ground truth for the 20-case detection accuracy test --- events
  that were detected are always in the log, regardless of what the consumer did. \\[0.5em]

\textbf{Geospatial Masking} &
  NFR4 -- Privacy &
  The Social module queries users by region but controls how much location detail it returns
  based on the ConsentState between two users: no consent returns only a city label; city-level
  consent returns a bounding box; mutual precise consent returns coordinates. The enforcement is
  at the query layer, not the UI --- the API never returns more than the consent level allows,
  even if a client sends a crafted request. \\

\bottomrule
\end{longtable}

\subsubsection{Pattern 1 --- Strategy (Booking Provider Integration)}

\begin{tabularx}{\textwidth}{@{}lX@{}}
\toprule
\textbf{Category} & Structural \\
\textbf{Problem} & Each booking provider (airline, hotel, transport) has a different API and

\subsubsection{Pattern 2 --- Chain of Responsibility (Disruption Suggestions)}

\begin{tabularx}{\textwidth}{@{}lX@{}}
\toprule
\textbf{Category} & Behavioural \\
\textbf{Problem} & Generating useful suggestions for a disruption is not a single deterministic
  function. Simple cases can be handled by rules; complex ones need an LLM; if both fail, the
  user should still get something useful. \\
\textbf{Why chosen} & The chain can be extended with new handlers without modifying existing
  ones. Handlers degrade gracefully when earlier ones succeed --- the LLM is not called if the
  rule engine already produced suggestions. \\
\textbf{Where used} & \texttt{src/modules/disruption/suggestions/handlers/} ---
  \texttt{RuleBasedHandler} $\to$ \texttt{LLMHandler} $\to$ \texttt{FallbackHandler}. \\
\bottomrule
\end{tabularx}

\vspace{0.5em}
The chain works as follows: \texttt{RuleBasedHandler} checks the disruption type and severity
against a lookup table and produces suggestions if it finds a match. \texttt{LLMHandler} calls
the Gemini API with the event context if the rule engine returned nothing. \texttt{FallbackHandler}
returns a generic ``check with your airline'' message if the LLM is unavailable or rate-limited.
Each handler calls \texttt{this.nextHandler.handle(ctx)} if it cannot resolve the request.

\subsection{Architecture Analysis}

\subsubsection{Implemented Architecture: Event-Driven Modular Monolith}

WanderSync is one NestJS process with six modules. Within that process, the disruption path is
fully event-driven: a CRON poller detects events, an in-process EventBus fans them out, and an
SSE stream pushes them to connected browsers. The rest of the system (auth, itinerary, booking,
notification, social) communicates synchronously through the REST API.

\subsubsection{Comparison Architecture: REST-Pull Polling Monolith}

\begin{longtable}{@{}L{2.8cm}L{5.6cm}L{5.6cm}@{}}
\toprule
\textbf{Dimension} & \textbf{Event-Driven SSE} & \textbf{REST-Pull Polling} \\
\midrule
\endhead
Notification speed &
  \textcolor{successgreen}{71 ms mean} --- instant from the user's perspective. &
  \textcolor{warningorange}{15 s mean} --- meaningful delay for time-critical travel alerts. \\[0.4em]
Server state &
  \textcolor{warningorange}{Stateful} --- one RxJS Subject per active user held in memory. &
  \textcolor{successgreen}{Stateless} --- no per-user objects; trivially scalable. \\[0.4em]
DB load from delivery &
  \textcolor{successgreen}{Zero} --- events pushed from memory. &
  \textcolor{warningorange}{Linear with users} --- each poll is a DB read. \\[0.4em]
Horizontal scaling &
  \textcolor{warningorange}{Needs sticky sessions} or Redis Pub/Sub to distribute events
  across instances. &
  \textcolor{successgreen}{Any load balancer} routes stateless polls freely. \\[0.4em]
Complexity &
  \textcolor{warningorange}{Higher} --- RxJS Observables, SSE lifecycle, dedup keys, per-user
  routing. &
  \textcolor{successgreen}{Lower} --- one REST controller and a DB query. \\[0.4em]
Right choice when &
  Real-time alerts are the product's core value (travel, finance, emergencies). &
  Information updates slowly and a short delay is acceptable (analytics, admin dashboards). \\

The event-driven architecture is the right choice for WanderSync because disruption alerts are
not informational --- they are actionable. A traveller who sees a cancellation in 71 ms can
rebook before the queue forms. A traveller who waits 30 seconds cannot.

% SECTION 7 — WHAT WE LEARNT
% ================================================================
\section{What We Learnt}

\textbf{1. Deduplication needs two layers, not one.}
We started with just a Redis TTL cache to prevent duplicate disruption events. It worked on
Linux but failed silently on Windows native Redis, which does not support the \texttt{SET NX EX}
compound command. We added a database guard as a second layer. Now the Redis layer is the fast
path and the database is the correctness guarantee. Either one failing alone does not cause
duplicate alerts.

\vspace{0.5em}
\textbf{2. In-process EventBus is faster and simpler than Redis Streams for SSE.}
The original design routed events through Redis Streams all the way to the SSE delivery path.
The extra serialisation round-trip added about 150 ms and made the flow harder to trace.
We moved event dispatch to an in-process RxJS Subject. Delivery latency dropped from $\sim$200 ms
to $\sim$71 ms, and the code became a lot easier to follow.

\vspace{0.5em}
\textbf{3. API contract mismatches are dangerous when the UI is optimistic.}
We renamed a field from \texttt{providerKey} to \texttt{provider} in the backend but did not
update the frontend DTO at the same time. The frontend's optimistic update made the booking
appear in the UI immediately, but the backend was silently rejecting the request. The booking
vanished on the next page refresh. We did not catch this for two days. Keeping a shared
TypeScript types file between frontend and backend would have caught it at compile time.

\vspace{0.5em}
\textbf{4. Chain of Responsibility is genuinely useful for graceful degradation.}
When the Gemini API key is missing or rate-limited, the suggestion chain falls through to the
rule-based handler and then to the fallback. The user always gets something. We did not have to
write a single \texttt{try/catch} in the caller --- the chain handles it transparently. The
pattern paid for itself on the first external API outage during testing.

\vspace{0.5em}
\textbf{5. Optimistic UI without rollback is a real risk.}
The \texttt{handleRemoveBooking} and \texttt{handleDelete} functions in the Itinerary page
update the local state before the backend call completes and do not roll back on failure.
If the request errors, the UI shows a state that does not match the database. We left this as
known debt for the prototype, but it would be the first thing we fixed in a production build.

\vspace{0.5em}
\textbf{6. Modular architecture makes parallel development much easier.}
Because each team member owned a bounded module with a clear interface, we almost never
blocked each other. The Auth module published its types; the Itinerary module consumed them.
The Disruption module published events; the Notification module subscribed. We were able to
work on four different features simultaneously with very few merge conflicts.
