\section{Overview}
% ================================================================

Two design patterns are applied in WanderSync to address recurring structural problems that
cannot be resolved by architectural tactics alone. Each pattern is documented with its category,
the specific problem it solves within the system, the classes and modules involved, and UML
diagrams showing the structure and runtime behaviour.

% ================================================================
\section{Pattern 1: Strategy}
% ================================================================

\begin{mdframed}[backgroundcolor=patternbox, linecolor=accent, linewidth=0.8pt]
\patternfield{Category}{Behavioral}
\patternfield{NFRs addressed}{NFR5 (Modifiability -- adding a new travel provider requires changes to at most 2 files outside the adapter module); NFR1 (Performance -- strategies are independently cacheable and testable with mock clients)}
\end{mdframed}

\vspace{0.5em}

\subsubsection{Problem}

The Booking Aggregation Service must retrieve and normalise data from multiple heterogeneous
travel providers — airlines, hotel platforms, and local transport services — each with a
distinct API and response schema. Without a formal pattern, provider-specific parsing logic
would proliferate inside the core aggregation service, coupling it to every provider's
implementation details. Every new provider addition would require modifying the
\texttt{BookingAggregationService} itself, violating the Modifiability NFR (NFR5) and GRASP
High Cohesion. Furthermore, individual provider interactions cannot be tested in isolation
because the fetch logic is entangled with the aggregation logic.

\subsubsection{Solution and Class Mapping}

The Strategy pattern defines a family of interchangeable algorithms (one per booking provider),
encapsulates each behind a shared interface, and allows the context (\texttt{BookingAggregationService})
to select and invoke the appropriate strategy at runtime without knowing the concrete type.

\begin{itemize}[noitemsep]
  \item \textbf{\texttt{IBookingStrategy}} -- interface declaring
        \texttt{fetchAndNormalize(userId: String): List<BookingRecord>}. All provider
        strategies implement this contract.
  \item \textbf{\texttt{BookingRecord}} -- canonical data object produced by every strategy;
        carries \texttt{bookingId}, \texttt{provider}, \texttt{BookingType},
        \texttt{departureTime}, and \texttt{arrivalTime}.
  \item \textbf{\texttt{FlightBookingStrategy}} -- encapsulates all flight-API calls and maps
        the airline-specific response schema to \texttt{BookingRecord}.
  \item \textbf{\texttt{HotelBookingStrategy}} -- encapsulates hotel-platform API calls and
        schema mapping.
  \item \textbf{\texttt{TransportBookingStrategy}} -- encapsulates local-transport API calls
        and schema mapping.
  \item \textbf{\texttt{ProviderRegistry}} -- maps provider-key strings (e.g.,
        \texttt{"flight"}, \texttt{"hotel"}) to \texttt{IBookingStrategy} instances;
        populated at application startup.
  \item \textbf{\texttt{BookingAggregationService}} -- context class; iterates all registered
        strategies via \texttt{ProviderRegistry}, invokes \texttt{fetchAndNormalize()}, and
        merges the results into a single unified list for the Itinerary Management Service.
\end{itemize}

Adding a new travel provider requires: (1) one new concrete strategy class implementing
\texttt{IBookingStrategy}; (2) one registration entry in \texttt{ProviderRegistry}. Zero
changes to \texttt{BookingAggregationService} or any other service, satisfying NFR5.

Note: the disruption pipeline (ADR-001, Tactic 4) uses a distributed Redis Streams
Pub/Sub architecture — a messaging construct, not the in-memory GoF Observer pattern.
These are categorically distinct; the Strategy pattern here operates synchronously within
a single process boundary.

\section{Pattern 2: Chain of Responsibility}
% ================================================================

\begin{mdframed}[backgroundcolor=patternbox, linecolor=accent, linewidth=0.8pt]
\patternfield{Category}{Behavioral}
\patternfield{NFRs addressed}{NFR3 (Security -- 100\% of endpoints require JWT enforcement, HTTPS); NFR5 (Modifiability -- handlers are independently testable and reorderable)}
\end{mdframed}

\vspace{0.5em}

\subsubsection{Problem}

Every inbound request to the API Gateway must pass through multiple independent validation steps
in a defined order: rate-limit check, JWT authentication, request schema validation, and audit
logging. Encoding all these steps as a flat conditional block in a single gateway controller
method produces a God Method that violates High Cohesion (GRASP) and makes each step
impossible to test in isolation. Furthermore, steps must be reorderable (e.g., placing the
audit logger after auth so that only authenticated requests are logged) and individually
enable/disable-able in test environments.

\subsubsection{Solution and Class Mapping}

The Chain of Responsibility pattern organises each processing step as an independent handler
that either processes the request and passes it to the next handler or short-circuits with an
error response.

\begin{itemize}[noitemsep]
  \item \textbf{\texttt{RequestHandler}} -- abstract base class declaring
        \texttt{setNext(handler)}, the protected \texttt{next} reference, and the abstract
        \texttt{handle(request: ApiRequest): ApiResponse} method.
  \item \textbf{\texttt{RateLimitHandler}} -- checks the caller's request count against a
        Redis counter; returns \texttt{429 Too Many Requests} if exceeded.
  \item \textbf{\texttt{AuthHandler}} -- validates the JWT signature and 24-hour expiry;
        returns \texttt{401 Unauthorized} if invalid, satisfying NFR3.
  \item \textbf{\texttt{SchemaValidationHandler}} -- validates the request body against the
        declared schema; returns \texttt{400 Bad Request} on violation.
  \item \textbf{\texttt{AuditLogHandler}} -- writes request metadata (user ID, endpoint,
        timestamp) to the audit log; always passes through to the next handler.
  \item \textbf{\texttt{GatewayController}} -- assembles the chain at startup:
        \texttt{RateLimitHandler $\to$ AuthHandler $\to$ SchemaValidationHandler $\to$
        AuditLogHandler $\to$ route dispatch}. Only this class changes when the chain order
        is modified.
\end{itemize}
